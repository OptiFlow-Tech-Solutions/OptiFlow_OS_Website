# API-001: API Versioning Prefix
- **Feature ID:** API-001 | **Category:** API | **Priority:** MEDIUM | **Phase:** 08 | **Status:** Pending | **Est. Hours:** 1

## Business Goal
Enable safe API evolution by versioning endpoints, allowing future breaking changes without disrupting existing integrations.

## Technical Goal
Prefix all API endpoints with `/api/v1/` instead of the current unversioned `/api/` paths.

## Problem Statement
The form submission endpoint is exposed at `/api/form-submit` with no version prefix. Introducing a breaking change to the request/response contract would silently break all existing clients, as there is no mechanism to route requests based on API version.

## Current Behavior
- `form-submit.js` handles requests at `/api/form-submit`
- `core.js` posts form data to `/api/form-submit`

## Expected Behavior
- `form-submit.js` handles requests at `/api/v1/form-submit`
- `core.js` posts form data to `/api/v1/form-submit`
- The old `/api/form-submit` endpoint either redirects or returns a `410 Gone` with a deprecation notice

## Affected Pages / Components / Files
- `api/form-submit.js` — update route path
- `assets/js/core.js` — update fetch URL

## Dependencies
None

## Acceptance Criteria
- [ ] All API endpoints are served under `/api/v1/`
- [ ] Client-side requests target the versioned endpoints
- [ ] Old unversioned paths return an appropriate migration response (301 redirect or 410 Gone with message)
- [ ] `npm run build && npm run validate` passes

## Definition of Done
- [ ] Code implemented
- [ ] Tests passing
- [ ] Validation passing

## Implementation Notes
Add a version prefix constant to both files so future version bumps are single-line changes. For backward compatibility during transition, add a redirect from `/api/form-submit` to `/api/v1/form-submit` that can be removed in a future version.

## Rollback Notes
Restore the original `/api/form-submit` paths and remove the version prefix.
