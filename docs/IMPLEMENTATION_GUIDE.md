# OptiFlow OS Website — Implementation Guide

**Version:** 1.0.0 | **Date:** 2026-07-09 | **Status:** Pre-Implementation

## Purpose

This guide defines HOW to implement each vertical slice feature from `WEBSITE_FEATURE_INVENTORY.md`. It establishes the standard workflow, conventions, and quality gates that every feature must pass before being considered complete.

---

## 1. Implementation Workflow (Per Feature)

### Step-by-Step

```
1. READ → Study the feature spec in WEBSITE_FEATURE_INVENTORY.md
2. PLAN → Break into frontend/backend/database/test subtasks
3. BRANCH → Create feature branch: feature/F-XXX-name
4. SCAFFOLD → Create empty files and folder structure first
5. TDD → Write failing test → implement → refactor → green
6. INTEGRATE → Wire frontend to backend to database
7. REVIEW → Self-review against Definition of Done
8. TEST → Run full test suite (unit + integration + E2E)
9. DOCUMENT → Update relevant docs if anything changed
10. PR → Open pull request with traceability link
11. MERGE → After CI passes + code review approved
12. VERIFY → Deploy to staging, smoke test
```

### Branch Naming Convention

```
feature/F-ARCH-001-project-setup
feature/F-DS-001-design-system
feature/F-MIG-HOME-001-home-page
```

### Commit Convention

```
feat(ARCH-001): initialize React + Django project scaffold
feat(DS-001): port core.css variables to design tokens
fix(HOME-001): correct typewriter animation timing on mobile
test(HOME-001): add visual regression test for hero section
docs: update SETUP.md with new port configuration
```

## 2. Vertical Slice Structure

Every feature is a COMPLETE vertical slice. Each feature directory contains:

```
feature/F-XXX-name/
├── frontend/           # React components + hooks + styles
├── backend/            # Django models + views + serializers + urls
├── database/           # Migration files (auto-generated by Django)
├── tests/
│   ├── unit/           # Frontend component tests + backend function tests
│   ├── integration/    # API endpoint tests
│   └── e2e/            # Playwright user flow tests
└── docs/               # Feature-specific docs (if needed)
```

### Definition of Done (Per Feature)

A feature is DONE only when ALL of these are true:

- [ ] Frontend components implemented and responsive
- [ ] Backend views/services implemented
- [ ] Database migrations created and applied
- [ ] API endpoints working (happy path + error cases)
- [ ] Input validation on both client and server
- [ ] Rate limiting applied to public endpoints
- [ ] Authentication/authorization enforced (if admin)
- [ ] Accessibility: WCAG 2.2 AA compliant (0 axe violations)
- [ ] Dark mode support verified
- [ ] Unit tests written (80%+ coverage on new code)
- [ ] Integration tests written (all API endpoints tested)
- [ ] E2E tests written (critical user flows)
- [ ] No linting errors (ESLint + stylelint + ruff)
- [ ] No TypeScript errors (tsc --noEmit)
- [ ] Visual parity with original static site (if migration)
- [ ] Error states handled (loading, empty, error, edge cases)
- [ ] Feature traceability documented in FEATURE_TRACEABILITY.md
- [ ] PR reviewed by at least one other developer
- [ ] CI pipeline passes all gates

## 3. Coding Standards

### Frontend (React + TypeScript)

- **Components:** One component per file. PascalCase filenames.
- **Hooks:** camelCase. Prefix with `use`. One hook per file.
- **Styles:** CSS Modules. Colocate with component. Import as `styles`.
- **Imports:** React first → third-party → local. Group by type.
- **Props:** Always typed with interface. Export interface.
- **State:** Prefer `useState` for local, Context for shared.
- **Effects:** Minimize `useEffect`. Prefer event handlers.
- **Error boundaries:** Wrap every page. Show fallback UI.

```typescript
// Component pattern
import { useState } from 'react';
import styles from './Button.module.css';

interface ButtonProps {
  variant: 'primary' | 'secondary';
  size?: 'sm' | 'md' | 'lg';
  children: React.ReactNode;
  onClick?: () => void;
  disabled?: boolean;
}

export const Button: React.FC<ButtonProps> = ({
  variant,
  size = 'md',
  children,
  onClick,
  disabled = false,
}) => {
  const [isPressed, setIsPressed] = useState(false);

  return (
    <button
      className={`${styles.btn} ${styles[variant]} ${styles[size]}`}
      onClick={onClick}
      disabled={disabled}
      aria-pressed={isPressed}
    >
      {children}
    </button>
  );
};
```

### Backend (Django + DRF)

- **Models:** One model per app. Use Django conventions.
- **Serializers:** Colocate with views. DRF ModelSerializer preferred.
- **Views:** Use DRF ViewSets for CRUD. Function-based for custom.
- **URLs:** One urls.py per app. Include in project urls.py.
- **Services:** Business logic in services.py. Keep views thin.
- **Permissions:** Custom permission classes. One file per app.
- **Tests:** One test file per module. pytest style.

```python
# Model pattern
from django.db import models

class DemoBooking(models.Model):
    STATUS_CHOICES = [
        ('PENDING', 'Pending'),
        ('CONFIRMED', 'Confirmed'),
        ('COMPLETED', 'Completed'),
        ('CANCELLED', 'Cancelled'),
    ]

    name = models.CharField(max_length=100)
    company_name = models.CharField(max_length=200)
    email = models.EmailField()
    mobile = models.CharField(max_length=15)
    team_size = models.CharField(max_length=20)
    industry = models.CharField(max_length=100)
    preferred_date = models.DateField()
    preferred_time_slot = models.CharField(max_length=20)
    status = models.CharField(max_length=20, choices=STATUS_CHOICES, default='PENDING')
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ['-created_at']
        indexes = [
            models.Index(fields=['preferred_date', 'preferred_time_slot']),
        ]
```

### Database (PostgreSQL)

- **Conventions:** See DATABASE.md §Conventions
- **Migrations:** Always generated by `makemigrations`. Review before commit.
- **Indexes:** Add for all FK columns and frequent filter/sort columns.
- **Timestamps:** `created_at` + `updated_at` on every table.
- **Soft delete:** Use `is_active` boolean, never hard delete.

### API

- **Format:** See API_REFERENCE.md for response envelope.
- **Auth:** JWT Bearer for admin. Anonymous for public.
- **Pagination:** `?page=1&page_size=20` (default 20).
- **Filtering:** `?status=X&source=Y&ordering=-created_at`.
- **Errors:** Always return structured error with code + message + fields.
- **Rate limiting:** Configurable per endpoint (see F-SEC-FORM-001).

## 4. Testing Standards

### Testing Pyramid (Per Feature)

```
    /  E2E (1-3 scenarios per feature)
   /   Integration (3-5 per API endpoint)
  /    Unit (5-15 per component/hook/function)
```

### Unit Tests

- **Frontend:** Vitest + React Testing Library. Render component, assert behavior.
- **Backend:** pytest. Test models, services, serializers, validators.

### Integration Tests

- **Frontend:** MSW for API mocking. Test form submission, search, navigation.
- **Backend:** pytest-django. Test API endpoints with test database.

### E2E Tests

- **Playwright:** One spec per critical flow. Cross-browser (4 projects).
- **Visual regression:** Screenshot comparison with original static site.

### Coverage Threshold

| Layer | Minimum |
|-------|---------|
| Frontend statements | 80% |
| Backend lines | 80% |
| Critical paths | 100% |

## 6. Environment Configuration

See `ENVIRONMENT.md` for all variables.

### Development

```bash
cp .env.example .env
# Set DJANGO_DEBUG=True
# Set local database credentials
docker compose up --build
```

### Adding New Environment Variables

1. Add to `backend/config/settings.py` (read from `os.environ`)
2. Add to `ENVIRONMENT.md` with description + default
3. Add to `.env.example` with placeholder value
4. Add to `docker-compose.yml` under `environment:` for relevant service
5. Never commit real values

## 7. Feature Dependencies

### Blocking Dependencies

- **F-ARCH-001** blocks ALL other features.
- **F-AUTH-001** blocks all admin features (F-ADMIN-001, F-LEAD-001, etc.).
- **F-DS-001** should be implemented before any page migration to ensure consistent components.

### Implementation Order

Follow WEBSITE_FEATURE_INVENTORY.md Part G (Recommended Implementation Order).

### Avoid Circular Dependencies

- UI layer depends on API layer, never vice versa.
- API layer depends on service layer, never vice versa.
- Service layer depends on models, never vice versa.

```
UI → API → Service → Model → Database
  ←────── NEVER ──────←
```

## 9. Common Pitfalls

1. **Visual drift:** Always compare against original static HTML. Never "improve" the design.
2. **API overfetching:** Return only what the UI needs. No nested objects unless requested.
3. **Missing error states:** Every component needs loading, error, and empty states.
4. **Hardcoded values:** Company info (phone, email, address) must come from site settings API.
5. **Inline styles:** Never use hex colors directly. Always use CSS variables from design tokens.
6. **No testing:** Features without tests are NOT complete. TDD is mandatory.
7. **Monolithic components:** Keep components small (under 200 lines). Split if growing.
8. **Missing accessibility:** Every interactive element must be keyboard-accessible and screen-reader friendly.
