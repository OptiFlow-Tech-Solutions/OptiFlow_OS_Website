# SEC-004: Phone Validation for Indian Numbers
- **Status:** Completed | **Priority:** HIGH | **Est. Hours:** —
- **Category:** Security | **Phase:** 03
- **Score Impact:** Security +3, Data Quality +4

## Problem
Phone validation used a permissive regex (`^\+?\d{7,15}$`) that accepted any 7–15 digit string, allowing garbage data into the CRM pipeline.

## Expected
Strict validation for Indian mobile numbers: optional `+91` country code (with optional space/hyphen), followed by a digit from `[9876]` (valid Indian mobile prefixes) and nine more digits.

## Affected
- `form-submit.js` (validation schemas)
- `contact.html` (client-side pattern attribute)
- `demo-booking.html` (client-side pattern attribute)

## Acceptance Criteria
- [x] Regex: `^(\+91[\s-]?)?[9876]\d{9}$`.
- [x] Valid numbers accepted: `9876543210`, `+919876543210`, `+91-9876543210`.
- [x] Invalid numbers rejected: `1234567890`, `+19876543210`, `98765432`.
- [x] Server-side validation matches client-side `pattern` attribute.

## Implementation
Updated the validation schema in `form-submit.js` and the `pattern` attribute on `<input type="tel">` elements in `contact.html` and `demo-booking.html`. All three use the same regex literal.

## Dependencies
None.

## Definition of Done
- [x] Deployed to production.
- [x] Unit tests pass for valid and invalid phone numbers.
- [x] Client-side HTML5 validation blocks invalid numbers before submission.
