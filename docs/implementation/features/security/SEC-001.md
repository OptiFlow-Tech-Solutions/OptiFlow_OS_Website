# SEC-001: Rate Limiting via Cloudflare KV
- **Status:** Completed | **Priority:** CRITICAL | **Est. Hours:** —
- **Category:** Security | **Phase:** 03
- **Score Impact:** Security +8

## Problem
Contact/demo form endpoints had no rate limiting, leaving them exposed to abuse, spam floods, and denial-of-service via repeated submissions.

## Expected
Per-IP rate limiting backed by Cloudflare KV with a 10-minute sliding window, max 5 requests per IP.

## Affected
- `form-submit.js` — replaced in-memory `globalThis._rateLimit` with KV-backed TTL storage.

## Acceptance Criteria
- [x] Each IP limited to 5 requests per 10-minute window.
- [x] Exceeding the limit returns HTTP 429.
- [x] KV TTL auto-expires entries; no manual cleanup needed.
- [x] Rate-limit check failure is non-blocking (fail-open): if KV is unreachable the request proceeds.

## Implementation
Stores request counts in a Cloudflare KV namespace keyed by IP. On each request the Worker reads the current count, increments, and writes back with a TTL matching the window. The old `globalThis._rateLimit` map was removed.

## Dependencies
None.

## Definition of Done
- [x] Deployed to production.
- [x] KV namespace bound in `wrangler.toml`.
- [x] Graceful degradation verified (KV outage does not block legitimate traffic).
