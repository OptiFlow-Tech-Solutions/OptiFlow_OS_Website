# ACC-003: ARIA Labels on Interactive Components
- **Status:** Pending | **Priority:** LOW | **Est. Hours:** 2.0
- **Category:** Accessibility | **Phase:** 05
- **Score Impact:** A11y +5

## Problem / Expected / Affected
Several interactive elements — buttons, form inputs, links acting as buttons — lack proper ARIA roles, labels, or `aria-labelledby` associations. This causes screen readers to announce generic "button" or "clickable" without meaningful context, making the site harder to operate for assistive-technology users.

Expected: All interactive elements have an appropriate accessible name via `aria-label`, `aria-labelledby`, or associated `<label>`. Form inputs are properly labelled. Buttons without visible text have descriptive `aria-label` attributes.

## Acceptance Criteria
- [ ] axe-core scan reports zero serious ARIA violations across all pages
- [ ] All `<input>`, `<select>`, and `<textarea>` elements have associated labels
- [ ] All `<button>` elements have either visible text or a descriptive `aria-label`
- [ ] Icon-only buttons (search, menu toggle, etc.) have `aria-label`
- [ ] Navigation, header, and footer included in the audit

## Implementation
Run an axe-core automated scan across every page. Triage each violation, fix the markup (add labels, roles, or `aria-labelledby` as appropriate), and re-scan until zero serious ARIA violations remain.

## Dependencies / Definition of Done
- **Depends on:** ACC-001 (decorative SVG cleanup reduces scan noise)
- **DoD:** axe-core scan across all 14 pages reports zero serious or critical ARIA violations.
