# database-data-management Specification

## Purpose
Define structured data management for Cloudflare KV-backed application data including data schemas, key naming conventions, lifecycle policies, admin export, pagination, and audit logging for the OptiFlow OS website.

## Requirements

### Requirement: KV Key Naming Convention

The system SHALL use structured, date-prefixed key names for all KV records to enable efficient listing and TTL-based lifecycle management.

#### Scenario: Submission key format
- **WHEN** a valid form submission is stored in KV
- **THEN** the key SHALL be `sub:{YYYY-MM-DD}:{uuid}`
- **AND** the UUID SHALL be a v4 UUID generated by the worker

#### Scenario: Notification log key format
- **WHEN** an email notification log is stored in KV
- **THEN** the key SHALL be `notif:{YYYY-MM-DD}:{uuid}`

#### Scenario: Audit entry key format
- **WHEN** an audit log entry is stored in KV
- **THEN** the key SHALL be `audit:{YYYY-MM-DD}:{uuid}`

#### Scenario: Newsletter subscriber key format
- **WHEN** a newsletter subscriber is stored in KV
- **THEN** the key SHALL be `subscriber:{sha256(email)}`
- **AND** this SHALL provide natural deduplication (same email = same key)

### Requirement: KV Value Schema Versioning

The system SHALL include a `schema` field in every KV value to enable forward-compatible deserialization and migration.

#### Scenario: Submission value schema
- **WHEN** a submission is stored in KV
- **THEN** the value SHALL include `"schema": "submission.v1"`

#### Scenario: Notification log value schema
- **WHEN** a notification log is stored in KV
- **THEN** the value SHALL include `"schema": "notification.v1"`

#### Scenario: Audit entry value schema
- **WHEN** an audit entry is stored in KV
- **THEN** the value SHALL include `"schema": "audit.v1"`

#### Scenario: Schema mismatch on read
- **WHEN** a KV value is read and its schema does not match the expected version
- **THEN** the worker SHALL log a warning with the key and actual schema
- **AND** the record SHALL be skipped (not returned to the caller)

### Requirement: Submission Value Schema

The system SHALL enforce a consistent JSON schema for submission KV values.

#### Scenario: Valid submission value structure
- **WHEN** a submission is stored in KV
- **THEN** the value SHALL be a JSON object with keys: `schema`, `id`, `timestamp`, `formName`, `fields`, `utm`, `ip`, `emailSent`, `spam`
- **AND** `schema` SHALL be `"submission.v1"`
- **AND** `id` SHALL be the same UUID used in the key
- **AND** `timestamp` SHALL be ISO-8601 format
- **AND** `formName` SHALL be one of `"contact"`, `"demo-booking"`, `"newsletter"`
- **AND** `fields` SHALL be an object with form-specific field names and values
- **AND** `ip` SHALL be a hashed IP (never raw)
- **AND** `spam` SHALL be a boolean

### Requirement: Notification Log Value Schema

The system SHALL enforce a consistent JSON schema for notification log KV values.

#### Scenario: Valid notification log structure
- **WHEN** a notification log is stored in KV
- **THEN** the value SHALL be a JSON object with keys: `schema`, `id`, `timestamp`, `type`, `to`, `subject`, `success`
- **AND** `schema` SHALL be `"notification.v1"`
- **AND** `id` SHALL be the same UUID used in the key

### Requirement: Audit Logging

The system SHALL write audit entries to KV for all data mutation operations and admin access events.

#### Scenario: Audit entry on submission write
- **WHEN** a form submission is accepted and stored in KV
- **THEN** an audit entry SHALL be written with `action: "submission_write"` and the submission key in `resource`

#### Scenario: Audit entry on submission export
- **WHEN** the admin export endpoint is accessed
- **THEN** an audit entry SHALL be written with `action: "submission_export"` and the actor's hashed IP

#### Scenario: Audit entry on admin login
- **WHEN** a successful admin login occurs
- **THEN** an audit entry SHALL be written with `action: "admin_login"`

#### Scenario: Audit entry value structure
- **WHEN** an audit entry is stored
- **THEN** the value SHALL include: `schema`, `id`, `timestamp`, `action`, `actor`, `resource`, `detail`

#### Scenario: KV write failure does not block audit
- **WHEN** audit KV write fails
- **THEN** the originating operation SHALL still succeed
- **AND** a console warning SHALL be logged

### Requirement: Admin Pagination

The admin submissions endpoint SHALL support offset-based pagination via query parameters.

#### Scenario: Default page
- **WHEN** GET `/api/admin/submissions` is called without page params
- **THEN** page 1 with 20 items per page SHALL be returned
- **AND** the response SHALL include `pagination: { page: 1, perPage: 20, total: N, pages: ceil(N/20) }`

#### Scenario: Specific page
- **WHEN** GET `/api/admin/submissions?page=2&per_page=10` is called
- **THEN** 10 items from the second page SHALL be returned
- **AND** `pagination.page` SHALL be 2, `pagination.perPage` SHALL be 10

#### Scenario: Out of range page
- **WHEN** GET `/api/admin/submissions?page=999` is called and 999 exceeds total pages
- **THEN** an empty submissions array SHALL be returned with correct pagination metadata

#### Scenario: Page capped to max
- **WHEN** GET `/api/admin/submissions?per_page=200` is called
- **THEN** `per_page` SHALL be capped to 100 (the maximum allowed)

### Requirement: Admin Data Export

The system SHALL expose an export endpoint at `/api/admin/submissions/export` that returns CSV or JSON formatted submission data.

#### Scenario: CSV export
- **WHEN** GET `/api/admin/submissions/export?format=csv` is called with valid Bearer token
- **THEN** the response SHALL have Content-Type `text/csv`
- **AND** the CSV SHALL include a header row with all field keys
- **AND** each submission SHALL be one row
- **AND** the Content-Disposition header SHALL be `attachment; filename="optiflow-submissions.csv"`

#### Scenario: JSON export
- **WHEN** GET `/api/admin/submissions/export?format=json` is called
- **THEN** the response SHALL be a JSON array of submission objects
- **AND** Content-Type SHALL be `application/json`
- **AND** the Content-Disposition header SHALL be `attachment; filename="optiflow-submissions.json"`

#### Scenario: Filtered export
- **WHEN** GET `/api/admin/submissions/export?format=csv&formName=contact` is called
- **THEN** only submissions with `formName: "contact"` SHALL be included in the export

#### Scenario: Unauthenticated export
- **WHEN** the export endpoint is called without a valid Bearer token
- **THEN** HTTP 401 SHALL be returned

### Requirement: Data Lifecycle Management

The system SHALL automatically purge expired records via a scheduled Worker.

#### Scenario: Submission TTL enforcement
- **WHEN** the scheduled cleanup worker runs
- **THEN** all `sub:` keys with timestamp older than 30 days SHALL be deleted from KV

#### Scenario: Notification TTL enforcement
- **WHEN** the scheduled cleanup worker runs
- **THEN** all `notif:` keys with timestamp older than 7 days SHALL be deleted from KV

#### Scenario: Audit TTL enforcement
- **WHEN** the scheduled cleanup worker runs
- **THEN** all `audit:` keys with timestamp older than 90 days SHALL be deleted from KV

#### Scenario: Cleanup audit log
- **WHEN** the cleanup worker completes a run
- **THEN** an audit entry SHALL be written with `action: "data_cleanup"` and `detail: { submissions_deleted: N, notifications_deleted: N, audit_deleted: N }`

### Requirement: Subscriber Deduplication

The system SHALL deduplicate newsletter subscribers by email address.

#### Scenario: New subscriber stored
- **WHEN** a newsletter signup is accepted
- **THEN** the subscriber SHALL be stored at key `subscriber:{sha256(email)}`
- **AND** the value SHALL include `{ email, subscribedAt, source }`

#### Scenario: Duplicate subscriber overwrite
- **WHEN** an existing subscriber email is submitted again
- **THEN** the existing KV key SHALL be overwritten with updated `subscribedAt`
- **AND** the older timestamp SHALL be preserved in `firstSeen`

### Requirement: KV Namespace Configuration

The system SHALL use separate Cloudflare KV namespaces for different data domains.

#### Scenario: Namespace binding
- **WHEN** Workers are deployed
- **THEN** the `SUBMISSIONS` namespace binding SHALL store form submissions
- **AND** the `NOTIFICATIONS` namespace binding SHALL store email notification logs
- **AND** the `AUDIT` namespace binding SHALL store audit log entries
