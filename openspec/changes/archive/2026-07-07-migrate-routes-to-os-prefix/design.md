## Context

The OptiFlow OS website is a static HTML site generated by a custom Node.js build script (`scripts/assemble.mjs`). There is no framework, no router, no dynamic routes. Every URL maps directly to a folder containing `index.html` inside `dist/`. All 14 product pages currently live at the root level (`/pricing/`, `/features/`, etc.).

The migration moves all product pages under `/os/` to reserve the root domain for a future corporate website. The architecture must remain scalable for additional products (`/crm/`, `/chat/`, `/analytics/`, `/api/`) without major future rework.

## Goals / Non-Goals

**Goals:**
- Serve all product pages under `/os/` prefix
- Provide a simple root `/` → `/os/` redirect, easy to remove later
- Update all internal links, navigation, SEO metadata, and infrastructure configs
- Preserve all existing functionality, design, and behavior
- Keep the build system's architecture clean for future product prefixes

**Non-Goals:**
- Adding a real router or framework
- Changing page content, design, or business logic
- Adding support for dynamic routes or SSR
- Creating a corporate homepage
- Moving assets under `/os/` — static assets stay at root (`/assets/`)
- Renaming pages or restructuring the page hierarchy

## Decisions

### Decision 1: Filesystem prefix over rewrite-based routing

**Chosen**: Modify `SRC_MAP` in `assemble.mjs` to output pages into `dist/os/` subdirectory.

**Alternative considered**: Keep pages at `dist/` root and use Nginx/Netlify rewrites to serve them under `/os/`. Rejected because:
- Sitemap and canonical URLs would need manual URL rewriting anyway
- Multiple layers of URL transformation increase debugging complexity
- Filesystem structure should match URL structure for clarity
- `serve` (used in `npm run dev`) doesn't support complex rewrites

### Decision 2: Plain HTML redirect at root

**Chosen**: A `dist/index.html` with `<meta http-equiv="refresh">` + JavaScript fallback.

**Alternative considered**: Nginx/Netlify 301 redirect. Rejected because:
- 301 is a permanent redirect, but this is temporary until the corporate site exists
- An HTML page gives us a landing surface if the corporate site needs a teaser first
- Simpler to remove: just delete the file (vs updating server configs)

### Decision 3: Error pages stay at root

**Chosen**: 404.html and 500.html remain at `dist/404.html` and `dist/500.html`.

**Rationale**: Error pages serve the entire domain, not just `/os/`. They should work for any URL pattern including future products. Netlify and Nginx already route to these by status code, not path.

### Decision 4: Assets remain at root

**Chosen**: `/assets/css/core.css`, `/assets/js/core.js`, `/assets/img/...` unchanged.

**Rationale**: Assets are shared across all product pages. Moving them under `/os/` would require duplicate copies for future products. The `/assets/` path is already cached with immutable headers in Nginx.

### Decision 5: Single SRC_MAP change, not per-page rewrites

**Chosen**: Update all 16 `SRC_MAP` keys to include `os/` prefix. The rest of the build script (page URL generation, sitemap, JSON-LD, manifest) derives URLs from these keys.

**Rationale**: The `pageInfo.file` field in `site.json` drives `SRC_MAP` lookup and URL generation. Changing the map keys cascades correctly through the entire build pipeline — sitemap URLs, canonical URLs, OG URLs, and JSON-LD are all computed from the same source.

## Risks / Trade-offs

- **[Risk]** Netlify trailing-slash redirects need updating. The existing 10 redirects (e.g., `/pricing` → `/pricing/`) must map to `/os/pricing` → `/os/pricing/`. Missing one leaves a broken redirect. → **Mitigation**: Validate script catches broken links post-migration.

- **[Risk]** Nginx trailing-slash rewrite regex only matched single-segment paths. With `/os/` prefix, paths become 2-segments. → **Mitigation**: Simplify the regex to match any depth, or just rely on `try_files $uri $uri/ $uri/index.html =404` which already handles trailing slashes via directory index.

- **[Risk]** Dev mode (`npm run dev` + `serve`) doesn't support the root redirect HTML. → **Mitigation**: The redirect page is a static HTML file — `serve` handles it naturally. The `start_url` in manifest.json is updated to `/os/`.

- **[Risk]** Bookmarked URLs break. Existing users and search engines with cached URLs hit the root pages that no longer exist. → **Mitigation**: Add 301 redirects in Netlify/Nginx from old paths to new `/os/...` paths for at least the major pages.

## Open Questions

- Should we add 301 redirects from old root paths (e.g., `/pricing/` → `/os/pricing/`) for SEO preservation? Recommended: yes, add to Netlify.
- Does the Plausible analytics `data-domain` config need updating? The domain stays `optiflow.in` — path changes shouldn't affect analytics. No change needed.
